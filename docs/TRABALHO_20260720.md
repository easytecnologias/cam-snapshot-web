## 2. Sistema de migrations

### O problema

O banco era criado com `CREATE TABLE IF NOT EXISTS` mais funções de remendo (`_ensure_*_schema`). Isso tem uma falha silenciosa:

- Num banco **vazio**, o `CREATE TABLE` define o schema final.
- Num banco **já existente**, o `CREATE TABLE` não faz nada, e o schema real passa a depender das funções de remendo.

Os dois caminhos divergiram em produção — e nada detectava. O sintoma concreto: a restrição `UNIQUE(ip)` antiga sobrevivia no banco de produção enquanto uma instalação nova nascia sem ela. Efeito prático: **dois clientes não podiam ter uma câmera com o mesmo IP.**

### O que foi feito

`app/core/migrations.py` — executor de migrations numeradas:

- Arquivos `.sql` em `migrations/<componente>/<backend>/<versão>_<nome>.sql`
- Cada migration aplicada fica registrada com checksum. Editar uma já aplicada é erro, não "atualização".
- Cada migration roda dentro de uma transação. Falhou, nada daquela versão fica.
- **Adoção**: banco pré-existente tem a versão base carimbada sem executar, e segue da próxima. É o que permitiu migrar produção sem recriar nada.

Migrations criadas:

| Componente | Versão | O que faz |
|---|---|---|
| main | 001 | Schema base (sites, câmeras, gravadores, canais) |
| main | 002 | Troca as chaves únicas globais por chaves por cliente |
| main | 003 | Tabela `olts` |
| auth | 001 | Schema base (tenants, usuários, tokens) |

### Verificação

`scripts/sightops_schema_migration_test.py` — prova que banco novo e banco antigo migrado terminam com schema idêntico.

Além disso, foi testado num **Postgres real** contra uma cópia do schema de produção, antes do deploy.

## 3. Isolamento entre clientes (multi-tenant)

As chaves únicas passaram de globais para por cliente:

| Tabela | Antes | Depois |
|---|---|---|
| `sites` | `UNIQUE(name)` | `UNIQUE(tenant_slug, name)` |
| `ip_cameras` | `UNIQUE(ip)` | `UNIQUE(tenant_slug, ip)` |
| `recorders` | `UNIQUE(source, host, port)` | `UNIQUE(tenant_slug, source, host, port)` |

A diferença importante: isso agora é **garantido pelo banco**, não por disciplina no código. Um endpoint novo que esqueça de filtrar por cliente esbarra na restrição em vez de misturar dados.

Verificado por `scripts/sightops_saas_smoke.py`.

## 4. Segurança das fotos

### O problema

Duas pastas inteiras (`/saida` e `/data/nvr_ai`) eram servidas como arquivos estáticos, **sem nenhuma separação por cliente**. Qualquer usuário logado de qualquer cliente conseguia ler arquivo de outro sabendo o nome.

### O que foi feito

- Os dois "mounts" foram removidos. Confirmado antes que nada no produto os consumia.
- As rotas de foto (`/data/snapshot`, `/data/dvr_snapshot`, `/data/nvr_snapshot`) passaram a exigir login e a resolver o arquivo **dentro da pasta do cliente do usuário**.
- Cabeçalhos de segurança adicionados nas respostas (nosniff, X-Frame-Options, Referrer-Policy, HSTS em HTTPS).

Verificado que a interface continua funcionando: o navegador manda o cookie de sessão nas tags `<img>` por serem da mesma origem.

## 5. Desempenho

### O problema

Toda função do módulo de autenticação começava chamando `init_auth_db()`. Depois das migrations isso ficou caro: no Postgres abre conexão, **pega um bloqueio global**, lê a tabela de migrations e checa o usuário padrão.

Como `get_user_by_token()` passa por ali, isso acontecia **uma vez por requisição autenticada** — e sendo bloqueio global, as requisições ficavam em fila.

A mudança de segurança das fotos amplificava: cada `<img>` virou uma requisição autenticada. Uma galeria de 30 câmeras pegava o mesmo bloqueio 30 vezes.

### O que foi feito

As 14 chamadas internas passaram a usar `_ensure_schema()`, que roda **uma vez por processo**.

**Medido:** 30 execuções em 30 requisições antes → **0 depois**.

## 8. Cadastro de OLT

> ⚠️ **Esta é a parte incompleta.** Ver [Pendências](#11-pendências).

### O problema

Não existia registro de OLT em lugar nenhum. As 8 operações de OLT recebiam IP, usuário e senha no corpo de cada requisição, e a lista de OLTs da tela de ONU era derivada das ONUs **já coletadas** — uma OLT nova só aparecia depois que alguém coletasse dela.

### O que foi construído

| Peça | Arquivo |
|---|---|
| Tabela `olts` | `migrations/main/{sqlite,postgres}/003_olt_registry.sql` |
| Criptografia da senha | `app/core/crypto.py` |
| Serviço (CRUD por cliente) | `app/services/olt_registry.py` |
| Rotas HTTP | `app/api/endpoints/olt.py` |
| Tela | `frontend/js/deployOlt.js` + seção no `index.html` |

Campos: nome, IP/host, **fabricante**, **modelo**, usuário, senha, conector, observações, ativa/inativa.

Fabricante e modelo são separados de propósito: hoje só existe Fiberhome (8820i e 4840e), mas o roteamento por modelo já existe no backend, e entrar Huawei/ZTE depois não deve exigir remodelar a tabela.

### Segurança da senha

- Guardada **cifrada** (`app/core/crypto.py`), nunca em texto puro.
- **Nunca volta pela API** — a leitura devolve apenas `has_password` (sim/não).
- Só `resolve_credentials()` decifra, e ela existe para ser chamada na hora de falar com o equipamento.
- Editar sem reenviar a senha **mantém a atual** (a tela nunca a recebe, então não teria como reenviar).

Verificado por dois testes independentes:
- `scripts/sightops_olt_registry_test.py` — camada de serviço: senha cifrada em disco, invisível na leitura, e um cliente sem acesso nenhum à OLT de outro (ler, editar, apagar ou decifrar).
- `scripts/sightops_olt_routes_test.py` — camada HTTP: a senha cadastrada **não pode aparecer no corpo de nenhuma resposta**.

## 9. Deploy

### Como era

Não havia repositório git no servidor. Os deploys eram feitos copiando arquivo por arquivo para dentro dos containers — em `/home/central` ficavam versões numeradas à mão (`sightops-app-140.js`, `sightops-index-140.html`). O branch local chegou a ficar **55 commits à frente** do GitHub, que por sua vez não refletia produção.

Pior: o frontend vivia **apenas na camada gravável do container**. Recriar o container apagaria o site inteiro, sem aviso.

### Como ficou

**Frontend em disco.** O `sightops-nginx` agora monta `/home/central/sightops-frontend-html` em `/usr/share/nginx/html`. Recriar o container não perde mais nada.

> Ressalva: o volume está num arquivo complementar (`/home/central/sightops-nginx-override.yml`), **não** no `docker-compose.platform.yml` (que é do root). Rodar `docker compose up` sem `-f /home/central/sightops-nginx-override.yml` desfaz a ligação. Os arquivos ficam salvos, mas o container volta a servir de dentro dele. Consertar de vez = adicionar o volume no compose do `/opt` como root.

**Backend por imagem.** `git push origin main` dispara o GitHub Actions, que publica `ghcr.io/easytecnologias/cam-snapshot-web:latest`. O servidor baixa e reinicia.

### O que aconteceu no deploy real

As migrations rodaram em produção pela primeira vez, exatamente como simulado:

```
main/postgres: banco pre-existente adotado nas versoes [1, 2]
main/postgres: aplicada 003_olt_registry
auth/postgres: banco pre-existente adotado nas versoes [1]
```

Nenhuma restrição removida, nenhum dado tocado. API subiu saudável, sem erros.

## 10. Erros cometidos no caminho

Registrados porque três deles quase custaram caro, e o padrão é o mesmo.

### Erro 1 — "bug crítico" que não existia

Foi reportado como **crítico** que o sistema de migrations não conseguiria criar a tabela de controle no Postgres, derrubando a API. A conclusão veio de análise das regras de conversão de tipo do PostgreSQL, **sem executar nada**.

Testado num Postgres 16 real: funciona perfeitamente. A regra de conversão estava mal compreendida.

**Lição:** nada deveria ser classificado como crítico sem ter sido executado. A dúvida devia ter rebaixado a severidade, não virar uma ressalva no rodapé.

### Erro 2 — "IDÊNTICO" que não comparou nada

Uma comparação de restrições entre dois bancos exibiu `>>> IDENTICO <<<`. Mas a consulta havia falhado nos dois lados (erro de tipo no SQL), gerando dois arquivos vazios — e comparar vazio com vazio passa.

Refeito com o tipo corrigido e com contagem de linhas provando que havia 8 registros de cada lado antes de confiar no resultado.

**Lição:** verificação que não prova ter olhado dados não vale como verificação.

### Erro 3 — o que quase derrubou produção

A tabela de controle das migrations usava a **versão** como chave, sem saber de qual componente era. Isso só funciona se `main` e `auth` viverem em bancos separados — suposição que não estava escrita em lugar nenhum, e que **a produção não cumpre** (`AUTH_DATABASE_URL` não é definida, então os dois usam o mesmo banco).

Efeito: a versão 1 do `auth` colidia com a versão 1 do `main`, e **a API não subiria após o deploy**.

Foi encontrado rodando a imagem já publicada contra um banco Postgres descartável, minutos antes de reiniciar a API. Nenhum teste cobria essa configuração — justamente a que produção usa.

**Lição:** o teste que faltava era o da configuração real, não o do caso ideal.

### Erro 4 — divergência de schema entre instalações

A migration base declarava as chaves únicas com `UNIQUE(...)` embutido no `CREATE TABLE`. No Postgres isso gera um nome automático que não casa com o nome procurado pela migration seguinte — resultado: instalação nova terminava com **índices duplicados**, e banco existente com um só.

Era exatamente a divergência que o sistema de migrations existe para impedir. Corrigido e verificado contra cópia do schema de produção.

### Erro 5 — senha exposta

Numa verificação no servidor, a máscara aplicada cobria apenas o formato de URL, não o formato `CHAVE=valor`. O valor de `DATABASE_PASSWORD` apareceu em texto claro na saída.

**Ação necessária:** trocar a senha do banco.

1. As 8 operações de OLT aceitarem `olt_id` e resolverem a senha no servidor
2. As telas de ONU e CFTV virarem lista suspensa

Além disso, a tela tem problemas identificados e ainda não corrigidos:

- **Os textos de exemplo parecem valores preenchidos.** Não existe regra de cor para placeholder no CSS do projeto, então "OLT Centro", "10.0.0.10" etc. saem quase da cor de um valor real.
- **O botão Salvar fica abaixo da dobra** — são 9 campos numa coluna de 300px.
- **Falta o campo Site** — existe no banco, não foi para a tela.
- **Falta testar conexão** — cadastra-se a OLT sem poder confirmar se IP e senha funcionam.

> **Antes de continuar:** está em aberto se o cadastro é mesmo o que se esperava do "menu OLT". A tela foi construída a partir de uma leitura do problema (senha redigitada a cada operação); confirmar se era essa a expectativa antes de investir nos passos 1 e 2.

### Melhorias conhecidas, sem urgência

#### Decisão futura: catálogo central de equipamentos

Não implementar agora, mas manter como evolução aprovada do SightOps: fabricantes e modelos não devem continuar fixos em cada tela. Criar futuramente um catálogo central, compartilhado por todos os menus de implantação e inventário.

Escopo previsto:

- categorias de equipamento: câmera, DVR/NVR, switch, roteador, access point, OLT, ONU/ONT e futuras categorias;
- fabricantes cadastráveis;
- modelos vinculados ao fabricante e à categoria;
- fluxo dependente `Categoria -> Fabricante -> Modelo`;
- capacidades técnicas por modelo, como canais, portas, PoE, PONs, GPON/EPON, protocolos e recursos;
- estado do modelo: ativo, inativo ou descontinuado;
- nível de integração: `suportado`, `parcial` ou `somente cadastro`;
- identificadores usados na descoberta automática por API, ONVIF, SSH ou SNMP;
- imagem, manual e observações técnicas opcionais.

Local sugerido para administração: `Configurações -> Catálogo de equipamentos`.

Regra importante: cadastrar um modelo no catálogo não significa que o SightOps possui driver para operar o equipamento. As telas devem usar o nível de integração para liberar ou bloquear testes, sincronização e configurações reais.

- A verificação de pós-condição da migration 002 confere que as chaves antigas sumiram, mas **não** que as novas existem. Um banco pela metade seria carimbado como pronto.
- `_commit`/`_rollback` engolem exceção silenciosamente — uma falha de commit viraria sucesso aparente.
- Os testes rodam em SQLite; produção é Postgres. Essa assimetria já custou dois bugs hoje.
- Jobs em memória (`ThreadPoolExecutor` + dicionário) não sobrevivem a reinício nem a mais de uma instância da API.

## 13. Roadmap futuro: MCP do SightOps

Decisão registrada para avaliação futura: criar um servidor MCP central do SightOps para permitir que assistentes de IA consultem e, posteriormente, operem o sistema por ferramentas padronizadas.

Exemplos desejados:

- consultar câmeras offline por site;
- listar ONUs com sinal ruim;
- consultar equipamentos que precisam de atenção;
- atualizar snapshots e testar OLTs;
- gerar relatórios operacionais;
- executar ações remotas somente com permissão e confirmação.

Fases sugeridas:

1. MCP somente leitura para inventário, monitoramento e diagnóstico;
2. relatórios, correlação de falhas e recomendações;
3. ações remotas com confirmação, RBAC e auditoria completa.

Regras obrigatórias antes de liberar mutações:

- isolamento rigoroso por tenant;
- autenticação vinculada ao usuário e cliente;
- permissões por função;
- confirmação para reiniciar, excluir, trocar IP ou provisionar equipamentos;
- trilha de auditoria de toda ação solicitada e executada;
- nenhuma ferramenta pode acessar dados de outro cliente.

Arquitetura pretendida: um MCP central no SaaS, usando os conectores MikroTik vinculados a cada tenant para alcançar redes remotas. Não será necessário instalar um MCP em cada cliente.

Responsabilidades continuam separadas:

- Zabbix: coleta e monitoramento contínuo;
- Grafana: séries históricas e visualizações;
- Conector: acesso seguro à rede remota;
- MCP: interface padronizada para a IA consultar e operar recursos do SightOps.

Este item está apenas documentado. Não deve ser implementado agora sem uma etapa específica de desenho de segurança, contratos das ferramentas e autorização.
